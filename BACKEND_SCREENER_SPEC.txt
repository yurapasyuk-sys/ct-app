# Техническое Задание: Backend API для Crypto Screener

## Обзор

Необходимо создать Rust backend, который будет собирать, хранить и отдавать данные по всем 535+ USDT Perpetual парам с Binance Futures. Frontend должен получать полный snapshot всех данных за < 100ms.

**Источник данных: Binance Futures Open API (публичный, без API ключей)**
- REST API: `https://fapi.binance.com/fapi/v1/`
- WebSocket: `wss://fstream.binance.com/`
- Документация: https://binance-docs.github.io/apidocs/futures/en/

### 2.2. Исторические данные (PostgreSQL)

#### Таблица: `klines`
Хранить свечи для расчёта метрик:

```sql
CREATE TABLE klines (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(20) NOT NULL,
    interval VARCHAR(5) NOT NULL,  -- '1m', '5m', '1h', '1d', '1w'
    open_time BIGINT NOT NULL,      -- Unix timestamp ms
    open DECIMAL(24, 8) NOT NULL,
    high DECIMAL(24, 8) NOT NULL,
    low DECIMAL(24, 8) NOT NULL,
    close DECIMAL(24, 8) NOT NULL,
    volume DECIMAL(24, 8) NOT NULL,
    quote_volume DECIMAL(24, 8) NOT NULL,
    trades_count INTEGER NOT NULL,
    taker_buy_volume DECIMAL(24, 8) NOT NULL,
    taker_buy_quote_volume DECIMAL(24, 8) NOT NULL,
    
    UNIQUE(symbol, interval, open_time)
);

-- Индексы для быстрых запросов
CREATE INDEX idx_klines_symbol_interval_time ON klines(symbol, interval, open_time DESC);
```

**Retention policy:**
- `1m` свечи: хранить 24 часа (1440 записей на символ)
- `5m` свечи: хранить 7 дней (2016 записей на символ)
- `1h` свечи: хранить 30 дней (720 записей на символ)
- `1d` свечи: хранить 365 дней (365 записей на символ)
- `1w` свечи: хранить 2 года (104 записи на символ)

#### Таблица: `open_interest_history`

```sql
CREATE TABLE open_interest_history (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(20) NOT NULL,
    timestamp BIGINT NOT NULL,       -- Unix timestamp ms
    open_interest DECIMAL(24, 8) NOT NULL,
    open_interest_value DECIMAL(24, 8) NOT NULL,  -- В USDT
    
    UNIQUE(symbol, timestamp)
);

CREATE INDEX idx_oi_symbol_time ON open_interest_history(symbol, timestamp DESC);
```

**Retention policy:** хранить 30 дней (720 записей на символ при часовом интервале)

#### Таблица: `funding_rate_history`

```sql
CREATE TABLE funding_rate_history (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(20) NOT NULL,
    funding_time BIGINT NOT NULL,    -- Unix timestamp ms
    funding_rate DECIMAL(18, 10) NOT NULL,
    mark_price DECIMAL(24, 8),
    
    UNIQUE(symbol, funding_time)
);

CREATE INDEX idx_funding_symbol_time ON funding_rate_history(symbol, funding_time DESC);
```

**Retention policy:** хранить 90 дней

## 3. Сбор данных с Binance

### 3.1. WebSocket Streams (real-time)

Подключиться к комбинированным стримам Binance Futures:

```
wss://fstream.binance.com/stream?streams=
    !ticker@arr              // Все тикеры (обновление ~1 сек)
    !markPrice@arr@1s        // Все mark prices (каждую секунду)
    !bookTicker              // Все book tickers (real-time)
```

**Важно:** Использовать один multiplexed connection для всех стримов.

### 3.2. REST API Polling

| Endpoint | Частота | Назначение |
|----------|---------|------------|
| GET /fapi/v1/exchangeInfo | 1 раз при старте, потом раз в час | Список всех символов |
| GET /fapi/v1/openInterest?symbol=X | Каждые 5 сек (батчами) | Текущий OI для каждого символа |
| GET /futures/data/openInterestHist | Каждый час | История OI для расчёта oi_change_8h |
| GET /fapi/v1/klines | Каждую минуту | Свечи для расчёта метрик |
| GET /fapi/v1/fundingRate | Каждые 8 часов | История funding rate |

### 3.3. Rate Limits

Binance Futures limits:
- 2400 request weight per minute
- Использовать batch endpoints где возможно
- Реализовать exponential backoff при 429 ошибках

## 4. Расчёт метрик на бекенде

### 4.1. Ticks 5m
```rust
// Сумма trades из последних 5 x 1m свечей
fn calculate_ticks_5m(klines_1m: &[Kline]) -> u64 {
    klines_1m.iter().rev().take(5).map(|k| k.trades_count).sum()
}
```

### 4.2. Change 5m (%)
```rust
fn calculate_change_5m(klines_1m: &[Kline]) -> f64 {
    if klines_1m.len() < 5 { return 0.0; }
    let open_5m_ago = klines_1m[klines_1m.len() - 5].open;
    let current_close = klines_1m.last().unwrap().close;
    ((current_close - open_5m_ago) / open_5m_ago) * 100.0
}
```

### 4.3. Volume 5m / 1h (USDT)
```rust
fn calculate_volume(klines_1m: &[Kline], minutes: usize) -> f64 {
    klines_1m.iter().rev().take(minutes).map(|k| k.quote_volume).sum()
}
```

### 4.4. Volatility 15m (%)
```rust
fn calculate_volatility(klines_1m: &[Kline]) -> f64 {
    let prices: Vec<f64> = klines_1m.iter().rev().take(15).map(|k| k.close).collect();
    if prices.len() < 2 { return 0.0; }
    
    // Логарифмические доходности
    let returns: Vec<f64> = prices.windows(2)
        .map(|w| (w[1] / w[0]).ln())
        .collect();
    
    // Стандартное отклонение
    let mean = returns.iter().sum::<f64>() / returns.len() as f64;
    let variance = returns.iter().map(|r| (r - mean).powi(2)).sum::<f64>() / returns.len() as f64;
    
    variance.sqrt() * 100.0
}
```

### 4.5. Vdelta 1h (Buy Volume - Sell Volume)
```rust
fn calculate_vdelta(klines_1m: &[Kline]) -> f64 {
    klines_1m.iter().rev().take(60).map(|k| {
        let buy_volume = k.taker_buy_quote_volume;
        let sell_volume = k.quote_volume - k.taker_buy_quote_volume;
        buy_volume - sell_volume
    }).sum()
}
```

### 4.6. OI Change 8h
```rust
fn calculate_oi_change_8h(oi_history: &[OIRecord]) -> f64 {
    let now = current_timestamp_ms();
    let eight_hours_ago = now - (8 * 60 * 60 * 1000);
    
    let current_oi = oi_history.last().map(|r| r.value).unwrap_or(0.0);
    let old_oi = oi_history.iter()
        .filter(|r| r.timestamp <= eight_hours_ago)
        .last()
        .map(|r| r.value)
        .unwrap_or(current_oi);
    
    current_oi - old_oi
}
```

### 4.7. Day/Week Open
```rust
fn get_day_open(klines_1d: &[Kline]) -> f64 {
    // Последняя дневная свеча = текущий день
    klines_1d.last().map(|k| k.open).unwrap_or(0.0)
}

fn get_week_open(klines_1w: &[Kline]) -> f64 {
    // Последняя недельная свеча = текущая неделя
    klines_1w.last().map(|k| k.open).unwrap_or(0.0)
}
```

## 10. Ответы на технические вопросы

### 1. Авторизация: НЕТ (пока)
- API публичный, без авторизации
- Данные не секретные (Binance public API)
- При необходимости добавить rate limiting по IP
- В будущем можно добавить API key для premium функций

### 2. Масштабирование: до 1000 одновременных пользователей
- In-memory cache справится легко
- Один Rust сервер потянет 10k+ RPS
- При росте — добавить Redis для кэша между инстансами
- WebSocket: до 10k одновременных подключений на один сервер

### 3. Деплой: Hetzner CPX32 (уже есть)
- **Сервер: Hetzner CPX32** (4 vCPU AMD, 8 GB RAM, 80 GB SSD)
- Сервер уже настроен, Copilot готов к работе
- PostgreSQL установить локально или использовать managed (Supabase/Neon бесплатно)
- Systemd service для автозапуска

```bash
# Systemd service file: /etc/systemd/system/screener-backend.service
[Unit]
Description=Screener Backend API
After=network.target postgresql.service

[Service]
Type=simple
User=www-data
WorkingDirectory=/opt/screener-backend
ExecStart=/opt/screener-backend/screener-backend
Restart=always
RestartSec=5
Environment=DATABASE_URL=postgresql://screener:password@localhost:5432/screener
Environment=RUST_LOG=info
Environment=PORT=8080

[Install]
WantedBy=multi-user.target
```

- Nginx reverse proxy для HTTPS (Let's Encrypt)
- Альтернатива: Docker Compose

### 4. Мониторинг: Простой (логи + health endpoint)
- Structured logging (tracing crate)
- Health endpoint `/api/v1/health`
- При необходимости — Sentry для error tracking
- Prometheus/Grafana — overkill для старта, добавить позже

### 5. База данных: PostgreSQL (достаточно)
- PostgreSQL 15+ с правильными индексами
- TimescaleDB — overkill для наших объёмов
- ~2.5M записей klines максимум (535 символов × 4645 записей)
- Партиционирование таблиц не нужно на старте
